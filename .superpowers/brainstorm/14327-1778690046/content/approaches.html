Virtualization approach for "Available to Download"

Only the large section needs DOM windowing. Which implementation fits best?

A

@tanstack/react-virtual (new dependency)

Headless virtual list hook — you control all the DOM, it only manages which items are in the viewport. Wraps the existing <Table>/<TableRow> rows with a fixed-height container + absolute positioning.

Pros

  • True DOM windowing — only ~20 rows in the DOM regardless of list size
  • Most widely used, actively maintained, TypeScript-native
  • Works with existing row markup — no UI redesign
  • Lightweight (~7 KB)

Cons

  • New npm dependency
  • Requires a fixed row height (or measured heights) — Table rows must be consistent
  • Absolute-positioned rows means the container needs a known height
B

Progressive rendering (no new dependency)

Render the first N rows immediately, then use an IntersectionObserver sentinel at the bottom of the list. When the sentinel becomes visible, append the next N rows. Pure React state — no library needed.

Pros

  • Zero new dependencies
  • Works with any row height — no measurement needed
  • Simple mental model: "load more as you scroll"
  • Keeps existing Table/TableRow markup unchanged

Cons

  • DOM still grows as you scroll — after loading all 2500 items, you're back to the same problem
  • Not true virtualization: items above the viewport are never recycled
  • Slower search results: filtering resets to page 1, but previously loaded items still in DOM until reset
C

CSS content-visibility: auto (no new dependency)

All rows stay in the DOM, but the browser skips rendering/layout for off-screen rows using CSS. Add content-visibility: auto; contain-intrinsic-size: auto 48px to each row.

Pros

  • Zero new dependencies, minimal code change
  • Browser handles everything — no JS virtualization logic
  • Works with variable row heights

Cons

  • All 2500 rows still mount in the DOM and React renders them — only paint/layout is skipped
  • Initial render of 2500 React nodes is still slow
  • Less browser support than A or B (Chromium 85+, limited Firefox support)
  • Not true virtualization — JS heap still holds all row data as components
My recommendation

Option A (@tanstack/react-virtual) is the right choice here. It's the only option that keeps a constant DOM size regardless of list length or scroll position. With 2500+ items, that's the key property needed. The dependency cost is low and it doesn't require changing the existing UI shape — just the rendering mechanism for the Available to Download section.
