# Design Principles

---

## Our user

The primary user of Paratext 10 is **Suroj** ‚Äî an expert Bible translator who hasn't spent much time with technology. Paratext may be his main reason for using a computer at all. He prefers the app to be simple and just work, and is not attached to the way things have always been done.

Design for Suroj: capable, purposeful, and unpretentious.

---

## Styling defaults

**Default to ShadCN UI styling.** Unless a custom style carries specific meaning, leave elements at their ShadCN defaults. Custom styles that don't communicate anything add maintenance burden and get in the way of future restyling.

- **Prefer vanilla over custom.** Default ShadCN styling can always be changed later at the component level. Ad hoc overrides are harder to track, harder to update, and harder to reason about.
- **Only style meaningfully.** Apply custom styling only when it communicates something specific ‚Äî hierarchy, state, emphasis, or brand. If you can remove it and nothing is lost, remove it.
- **Consistent styling aids parsing.** When elements share consistent styling, users can scan and understand the interface faster. Visual noise from unnecessary variation slows comprehension.
- **Consistency enables contrast.** Intentionally contrastive styling ‚Äî a bold highlight, an accent color, a size shift ‚Äî is a powerful tool for directing attention, but only works when the surrounding design is consistent. Reserve it for moments that matter.

> **Note on ShadCN component variation:** Different ShadCN components (e.g., command menu items, popovers, dropdown menus) may apply different default styles to visually similar elements. This is by design ‚Äî the components have different underlying semantics. If similar content looks different across contexts, verify whether the underlying components differ before applying overrides, and surface that information to the design team.

## Color

### Semantic color pairing

Always pair background utility classes with their `-foreground` counterparts. Never mix `text-secondary` on `bg-secondary` ‚Äî it creates low contrast.

| Class | CSS Variable | Use case |
| --- | --- | --- |
| `tw-text-secondary` | `--secondary` | Text **on** primary backgrounds |
| `tw-text-secondary-foreground` | `--secondary-foreground` | Text **on** secondary backgrounds |

**Examples:**

- Secondary card: `bg-secondary text-secondary-foreground`
- Primary button text: `bg-primary text-secondary`

### State changes and contrast

Desktop UIs carry many overlapping states: hover, focus, selected, active, unread. Each state change typically involves a color shift. When multiple states combine, contrast can be squeezed from both ends simultaneously.

For example: if hover lightens a card background while activation darkens the text color, both shifts reduce contrast at the same time. This compounds quickly, and dark mode tends to amplify the problem. Always verify contrast when combining two or more state-driven color changes.

## Editor toolbars and action buttons

### Button order

Cancel/discard goes on the **left**; confirm/accept goes on the **right**. This is the standard convention and must be consistent across all editors and dialogs.

### Destructive action icon

Use `Trash2` (not `X`) when a button discards or deletes content. `X` means "close" ‚Äî use it only when nothing will be lost. `Trash2` is from `lucide-react`.

### Button variant hierarchy

The discard button should use `variant="secondary"`, and the accept button `variant="default"` (filled). Avoid pairing `ghost` + `ghost` ‚Äî it removes visual hierarchy and makes the primary action unclear.

### ButtonGroup

Use `ButtonGroup` and `ButtonGroupSeparator` from `lib/platform-bible-react/src/components/shadcn-ui/button-group.tsx` rather than manual gap spacing. Group related buttons together. Nesting two `ButtonGroup`s inside a parent automatically adds `gap-2` between the groups.

### Icon-only buttons

Every icon-only button must have an `aria-label`. Screen readers cannot infer meaning from an icon alone. Use the localized string for the tooltip text when available.

## Keyboard shortcuts in tooltips

Render keyboard shortcut hints using the HTML `<kbd>` element, not plain text in parentheses. The `<kbd>` styling provides sufficient visual distinction on its own ‚Äî no parentheses needed.

**Do:** `Undo <kbd>‚åò</kbd><kbd>Z</kbd>`

**Don't:** `Undo (‚åòZ)`

If a shared `Kbd` component doesn't exist yet, style inline:

```tsx
<kbd className="tw-rounded tw-border tw-border-border tw-bg-muted tw-px-1 tw-py-0.5 tw-text-xs tw-font-mono">
  ‚åòZ
</kbd>
```

## Keyboard navigation and focus

All interactive UI must be keyboard navigable. Focus should be indicated by a focus ring by default, unless UX specifies otherwise for a specific component.

Be aware that the focus ring sometimes goes on a **container** rather than the interactive element itself. For example, when a text input field contains buttons, the focus ring wraps the entire container ‚Äî not each individual button.

See also: [Responsiveness](./responsiveness.mdx) for minimum width requirements and container query guidance.
