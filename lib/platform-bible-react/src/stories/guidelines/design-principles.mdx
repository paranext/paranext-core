# Design Principles

This document defines the baseline quality standard for Paratext 10's UI. It is the **Definition of Done** for interface work — acceptance criteria on individual Jira items should treat this as the baseline and add only what is specific to that item.

---

## Our user

The primary user of Paratext 10 is **Saroj** — an expert Bible translator who hasn't spent much time with technology. Paratext may be his main reason for using a computer at all. He prefers the app to be simple and just work, and is not attached to the way things have always been done.

Design for Saroj: capable, purposeful, and unpretentious.

---

## Styling defaults

**Default to shadcn/ui UI styling.** Unless a custom style carries specific meaning, leave elements at their shadcn/ui defaults. Custom styles that don't communicate anything add maintenance burden and get in the way of future restyling.

- **Prefer vanilla over custom.** Default shadcn/ui styling can always be changed later at the component or theme level. Ad hoc overrides are harder to track, harder to update, and harder to reason about.
- **Only style meaningfully.** Apply custom styling only when it communicates something specific — hierarchy, state, emphasis, or brand. If you can remove it and nothing is lost, remove it.
- **Consistent styling aids parsing.** When elements share consistent styling, users can scan and understand the interface faster. Visual noise from unnecessary variation slows comprehension.
- **Consistency enables contrast.** Intentionally contrastive styling — a bold highlight, an accent color, a size shift — is a powerful tool for directing attention, but only works when the surrounding design is consistent. Reserve it for moments that matter.

> **Note on shadcn/ui component variation:** Different shadcn/ui components (e.g., command menu items, popovers, dropdown menus) may apply different default styles to visually similar elements. This is by design — the components have different underlying semantics. If similar content looks different across contexts, verify whether the underlying components differ before applying overrides, and surface that information to the design team.

## Color

### Semantic color pairing

Always pair background utility classes with their `-foreground` counterparts. Never mix `text-secondary` on `bg-secondary` — it creates low contrast.

| Class                          | CSS Variable             | Use case                          |
| ------------------------------ | ------------------------ | --------------------------------- |
| `tw:text-primary-foreground`   | `--primary-foreground`   | Text **on** primary backgrounds   |
| `tw:text-secondary-foreground` | `--secondary-foreground` | Text **on** secondary backgrounds |

**Examples:**

- Secondary card: `bg-secondary text-secondary-foreground`
- Primary button text: `bg-primary text-primary-foreground`

### State changes and contrast

Desktop UIs carry many overlapping states: hover, focus, selected, active, unread. Each state change typically involves a color shift. When multiple states combine, contrast can be squeezed from both ends simultaneously.

For example: if hover lightens a card background while activation darkens the text color, both shifts reduce contrast at the same time. This compounds quickly, and dark mode tends to amplify the problem. Always verify contrast when combining two or more state-driven color changes.

## Dialogs, toolbars, and action buttons

### Button order

Cancel/discard goes on the **left**; confirm/accept goes on the **right**. This is the standard convention and must be consistent across all editors and dialogs.

### Destructive action icon

Use `Trash2` (not `X`) when a button discards or deletes content. `X` means "close" — use it only when nothing will be lost. `Trash2` is from `lucide-react`.

### Button variant hierarchy

In dialogs and popovers, the discard button should use `variant="secondary"` or `variant="ghost"`, and the accept button `variant="default"` (filled). Avoid pairing `ghost` + `ghost` — it removes visual hierarchy and makes the primary action unclear.

### Button labels

Button *labels* — the words, and which apply model they express — are covered in [Applying Changes](?path=/docs/guidelines-applying-changes--docs#microcopy). In short: avoid **OK** (users dismiss it without reading), and use a specific action verb so the label says what will happen. This page owns how dialog buttons **look and arrange** (order, variant, icon, above); Applying Changes owns what they **say**.

### ButtonGroup

Use `ButtonGroup` and `ButtonGroupSeparator` from `lib/platform-bible-react/src/components/shadcn-ui/button-group.tsx` rather than manual gap spacing. Group related buttons together. Nesting two `ButtonGroup`s inside a parent automatically adds `gap-2` between the groups.

### Icon-only buttons

Every icon-only button must have an `aria-label`. Screen readers cannot infer meaning from an icon alone. Use the localized string for the tooltip text when available.

## Tooltips

Tooltips are required for every icon button and any control without a visible label — including
toolbar items. Tooltips are supplementary: if the tooltip would only repeat the visible label,
omit it. Keep tooltip text short enough to scan at a glance.

For full tooltip guidance — when to use a tooltip vs. a popover, keyboard-shortcut formatting,
and patterns under consideration — see [Tooltips](./tooltips.mdx).

## Keyboard navigation and focus

All interactive UI must be keyboard navigable. Focus should be indicated by a focus ring by default, unless UX specifies otherwise for a specific component.

Be aware that the focus ring sometimes goes on a **container** rather than the interactive element itself. For example, when a text input field contains buttons, the focus ring wraps the entire container — not each individual button.

### Keyboard shortcut catalog

Every keyboard shortcut in the entire application — main process, bundled extensions, and this component library — is documented in one curated catalog, so there is a single place to see what shortcuts exist, what they do, and where they are handled in code. It lives in the main Platform.Bible Storybook under **Reference → Keyboard shortcuts**.

**Definition of Done:** when you add, change, or remove any keyboard shortcut — a `keydown`/`keyup` handler, an Electron accelerator or `before-input-event` branch, or a `useHotkeys` binding — you must add, update, or remove its entry in the single catalog data file `src/stories/keyboard-shortcuts.data.ts` as part of the same change. A shortcut change that leaves the catalog stale is not done.

See also: [Responsiveness](./responsiveness.mdx) for minimum width requirements and container query guidance.
