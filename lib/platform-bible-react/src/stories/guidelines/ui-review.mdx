# UI Review

*Written by Alex Mercado, UX*

A UI work item can only be moved to Done once its UX review property is set to `done` or `review not needed`. This article covers what that review looks like and how to get there.

---

## Before you request a review

- [ ] Acceptance criteria are checked off or explicitly declared in the Jira work item
- [ ] Jira UX review property is set to `needs review`
- [ ] A Storybook story or documentation page exists for the work
- [ ] The PR description links to any reference UI using specific file paths and [Storybook](https://paranext.github.io/paranext-core/platform-bible-react-storybook/) links
- [ ] A before/after video or screen recording is posted in the PR description or Jira work item

## Submitting the request

Post in a new thread in [#review-ux](https://ptb.discord.com/channels/892072317436448768/1451254233910476992) on the Paratext Discord. Include:

- A link to the **Jira work item** — full URL, e.g. `https://paratextstudio.atlassian.net/browse/PT-3901`, not just `PT-3901`
- A link to the **GitHub PR**
- A brief description of what was done

Also include a link to the Jira work item **in the PR description**, so it is traceable from the code change.

## Reference

### Jira UX review property

Every Jira work item has a **UX review** property, set during grooming:

| State | Meaning |
| --- | --- |
| `open` | Default — not yet assessed |
| `needs review` | Developer is ready for UX review |
| `done` | UX review completed and approved |
| `review not needed` | UX sign-off explicitly not required for this item |

### Acceptance criteria

Checkboxes on a Jira work item assert that the developer **did the thing** — not that it has been verified. Checking a box is a claim, not a sign-off.

Verification is the responsibility of:

| What | Who verifies | Where to post | Notes |
| --- | --- | --- | --- |
| Code quality | Code reviewer | New [#reviews](https://ptb.discord.com/channels/1064938364597436416/1369016160485511222) thread on Platform.Bible Discord | Review happens in Reviewable (built on [GitHub](https://github.com/paranext)) |
| User interface and experience | UX reviewer | New [#review-ux](https://ptb.discord.com/channels/892072317436448768/1451254233910476992) thread on Paratext Discord — one thread per issue | Subtasks receive a directional review; parent work items require full UX approval. |
| Functional behavior | QA / testing | New [#10-simple-testing](https://ptb.discord.com/channels/892072317436448768/1451251686709395548) or [#10-power-testing](https://ptb.discord.com/channels/892072317436448768/1451251722340139141) thread on Paratext Discord — one thread per item | Testing happens independently after the code is marked done. |

### Task management

Work items are sometimes split into subtasks for engineering reasons. Subtasks can be treated as **check-ins** rather than full approvals — the parent work item is still the unit that requires UX sign-off. Don't block subtask PRs waiting for full UX approval, but make sure the parent gets a complete review before it closes.

### Storybook deliverables

Every UI work item should have a **Storybook deliverable** — a story or documentation page that demonstrates the component or interaction. This makes the work reviewable, testable, and shareable without needing to run the full app.

### Linking to reference UI

When your PR description references other parts of the UI for comparison, provide **specific file paths and Storybook links** — not just names. This allows reviewers and AI-assisted review tools to actually locate and verify those references.

**Unhelpful:**
> "Adjusted padding to be consistent with the rest of the platform."

**Helpful:**
> "Adjusted padding to match the [SearchBar](https://paranext.github.io/paranext-core/platform-bible-react-storybook/?path=/docs/basics-searchbar--docs) (`lib/platform-bible-react/src/components/basics/search-bar.component.tsx`) and the [TabToolbar](https://paranext.github.io/paranext-core/platform-bible-react-storybook/?path=/docs/advanced-tabtoolbar--docs) (`lib/platform-bible-react/src/components/advanced/tab-toolbar/tab-toolbar.component.tsx`)."

### Getting UX approval

For UX to approve a UI change, one of the following is required:

- **Before/after videos** of the change (posted in the PR or Jira work item), or
- A **call with UX** to walk through the change together

If neither is provided, UX approval cannot be given.
