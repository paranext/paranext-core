# Rebase a branch or PR onto its base branch

Rebase a feature branch onto its base branch (usually `origin/main`), resolving `dist/` conflicts by discarding the feature branch's dist changes (they will be regenerated by the build), rebuilding `platform-bible-utils` and/or `platform-bible-react` if they changed, then committing and force-pushing with lease.

## Argument

Optional single argument `$ARGUMENTS`:

- **PR URL** (e.g. `https://github.com/paranext/paranext-core/pull/1234`)
- **PR number** (e.g. `1234`)
- **Branch name** (e.g. `feat/my-thing`)
- **Omitted** → operate on the currently checked-out branch.

## Steps

1. **Confirm working tree is clean.**

   ```bash
   git status --porcelain
   ```

   If there are uncommitted changes, stop and ask the user how to proceed. Do not stash silently.

2. **Resolve the target branch and switch to it (if an argument was given).**

   - Record the current branch: `CURRENT=$(git rev-parse --abbrev-ref HEAD)`.
   - If `$ARGUMENTS` is empty, set `TARGET=$CURRENT` and skip to step 3.
   - Otherwise, determine what the argument refers to:
     - If it matches a URL (`http(s)://…/pull/<n>`) or is purely numeric, treat it as a PR reference.
     - Else treat it as a branch name.
   - Ask the user to confirm the branch switch **before** performing it, showing `CURRENT → <resolved target>`. Only proceed on explicit confirmation.
   - Checkout:
     - PR reference: `gh pr checkout <arg>` (handles URL or number; sets upstream tracking).
     - Branch name: `git fetch origin <branch> && git checkout <branch>`.
   - After checkout, set `TARGET=$(git rev-parse --abbrev-ref HEAD)`.
   - If `TARGET` is `main`, stop — this command is only for rebasing feature branches.

3. **Pull latest changes for `TARGET` from origin (fast-forward only).**

   ```bash
   git fetch origin "$TARGET"
   ```

   - Compare local vs remote: `git rev-list --left-right --count HEAD...origin/$TARGET`.
   - If local is behind and not ahead: `git pull --ff-only`.
   - If diverged (both ahead and behind), stop and surface — do not silently resolve.
   - If local is only ahead or equal, skip the pull.

4. **Detect the base branch.**

   ```bash
   BASE=$(gh pr view --json baseRefName -q .baseRefName 2>/dev/null || echo "")
   ```

   - If `gh pr view` succeeds and returns a value, use that as `BASE`.
   - If it fails (no PR found), default `BASE=main`.
   - **If `BASE` is not `main`, stop and ask the user to confirm** before continuing. Show them the detected base and let them accept or override.

5. **Fetch the base branch.**

   ```bash
   git fetch origin "$BASE"
   ```

6. **Rebase onto `origin/$BASE`.**

   ```bash
   git rebase "origin/$BASE"
   ```

7. **Resolve conflicts in a loop until the rebase completes.**

   While `git status` reports a rebase in progress:

   - Identify conflicted paths: `git diff --name-only --diff-filter=U`
   - For every conflicted path **inside a `dist/` directory** (any depth — e.g. `lib/platform-bible-react/dist/...`, `lib/platform-bible-utils/dist/...`, `extensions/**/dist/...`):
     - Take the base version and stage it: `git checkout --ours -- <path> && git add <path>`
       - (During rebase, `--ours` refers to the branch being rebased onto, i.e. `origin/$BASE`. We are discarding the feature branch's dist changes because the build will regenerate them.)
   - If, after handling all `dist/` conflicts, **non-dist conflicts remain**, stop and surface them to the user with a short summary of each file. Do not guess resolutions.
   - Once all conflicts for the current step are resolved and staged, continue:

     ```bash
     git rebase --continue
     ```

   Repeat until the rebase reports success (no rebase in progress).

   If at any point you need to abort, use `git rebase --abort` and report to the user — do not leave the tree in a half-rebased state.

8. **Detect which internal packages changed on this branch.**

   Compute the branch's changed files against the base:

   ```bash
   git diff --name-only "origin/$BASE"...HEAD
   ```

   Set flags based on paths (ignore paths inside each package's own `dist/`):

   - `NEEDS_UTILS_BUILD=1` if any changed path starts with `lib/platform-bible-utils/` and is **not** inside `lib/platform-bible-utils/dist/`.
   - `NEEDS_REACT_BUILD=1` if any changed path starts with `lib/platform-bible-react/` and is **not** inside `lib/platform-bible-react/dist/`.

9. **Build changed packages — utils first, then react.**

   Build **only** the packages flagged above, in this order:

   ```bash
   # If NEEDS_UTILS_BUILD
   cd lib/platform-bible-utils && npm run build && cd ../..

   # If NEEDS_REACT_BUILD
   cd lib/platform-bible-react && npm run build && cd ../..
   ```

   If a build fails, stop and surface the error. Do not commit a broken build.

10. **Commit any regenerated dist (if there are changes).**

    ```bash
    git status --porcelain
    ```

    If `lib/platform-bible-utils/dist/` or `lib/platform-bible-react/dist/` (or any other regenerated artifacts) show changes, stage and commit them together:

    ```bash
    git add lib/platform-bible-utils/dist lib/platform-bible-react/dist
    git commit -m "chore: rebuild dist after rebase onto $BASE"
    ```

    If there are no changes, skip this step.

11. **Force-push with lease.**

    Because the branch was rebased, a normal push will be rejected. Use `--force-with-lease` (safer than `--force`) to avoid clobbering upstream work you don't know about:

    ```bash
    git push --force-with-lease origin "$TARGET"
    ```

    If the push is rejected because the lease check failed, stop and surface it — do not escalate to `--force`.

12. **Report.**

    Summarize:
    - Starting branch and (if switched) the branch that was checked out.
    - Detected base branch and whether user confirmation was requested.
    - How many conflicts were resolved and how many were in `dist/` vs elsewhere.
    - Which packages were rebuilt (utils, react, both, or none).
    - Whether a dist-rebuild commit was created.
    - The final `HEAD` SHA and the branch pushed.
